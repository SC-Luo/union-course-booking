# 工會課程預約系統上線前檢查清單 (Launch Checklist)

本文件供秘書處行政人員與開發維護人員於系統正式上線前進行逐項核對與驗收，確保系統功能、安全及穩定度皆達到生產環境要求。

---

## 1. 必要環境變數檢查

請至 Vercel 或是正式主機環境中確認以下環境變數均已設定，且名稱無誤（切勿印出或寫下實際密碼）：

- `[ ]` **ADMIN_PASSWORD**：後台秘書處登入密碼。
- `[ ]` **ADMIN_SESSION_SECRET**：後台 Cookie Session 簽署密鑰。
- `[ ]` **BOOKING_DATA_SOURCE**：必須設定為 `"firestore"`。
- `[ ]` **FIREBASE_PROJECT_ID**：Firebase 專案 ID。
- `[ ]` **FIREBASE_CLIENT_EMAIL**：Firebase 服務帳號的 Client Email。
- `[ ]` **FIREBASE_PRIVATE_KEY**：Firebase 服務帳號的 Private Key（需保留 `\n` 換行符號）。
- `[ ]` **GOOGLE_SHEETS_WEBHOOK_URL**：(選填) 公司帳號 GAS 同步的 Web App URL。
- `[ ]` **TEACHING_ACCESS_CODE**：(選填) 講師/助教登入所需的共用授課通行碼。

---

## 2. 後台登入檢查

- `[ ]` **未登入阻斷**：嘗試直接存取 `/admin`、`/admin/students`、`/admin/exports`。系統必須攔截並重導向至 `/admin/login`。
- `[ ]` **跳轉保留 (next query)**：重導向至登入頁後，網址中必須包含 `?next=/admin/students` 參數。
- `[ ]` **正確登入跳轉**：輸入正確的 `ADMIN_PASSWORD`，系統必須寫入 `admin_session` cookie 並成功轉跳回原先存取的路徑（例如：`/admin/students`）。
- `[ ]` **錯誤密碼提示** : 輸入錯誤密碼，系統應留在登入頁並顯示：「密碼不正確，請重新輸入。」且保留原 `next` 參數。

---

## 3. Firestore 連線檢查

- `[ ]` **連線健康度**：系統在 Production 環境下，如果資料來源為 `firestore`，若連線失敗或初始化異常必須**直接拋出 Error**，禁止默默 Fallback 回落讀寫本機的 `booking-data.json`，防止造成資料不一致與假成功。
- `[ ]` **寫入與讀取**：新增一筆測試預約，確認資料已正確寫入 Google Cloud Firestore 的對應 Collections（`reservations`），而非本機 JSON。

---

## 4. 學員端預約流程

- `[ ]` **首頁分流**：進入 `/`，確認只看見「我要預約課程」、「我的預約」、「第一次使用」三大入口卡片與預約提醒條，課程分類與日曆應顯著往上移。
- `[ ]` **我要預約**：點擊「查看課程」按鈕，頁面能成功平滑滾動至下方的課程列表區。
- `[ ]` **預約表單填寫**：選擇一門有餘額的課程，填寫學員「姓名」與「手機末三碼」，送出後應進入預約成功頁。
- `[ ]` **防重複預約**：同一個學員（同姓名+手機末三碼）預約同門課程的另一個時段，系統應提示「已有此課程預約，不可重複預約」。
- `[ ]` **額滿限制**：額滿的時段應顯示為「額滿」，且不能再點擊預約。
- `[ ]` **截止限制**：前一天 18:00 後，預約按鈕應被隱藏或無法預約。

---

## 5. 預約查詢與取消流程

- `[ ]` **查詢預約**：進入 `/booking/search`，輸入「姓名」與「手機末三碼」，必須能查詢到該員的所有有效預約。
- `[ ]` **自動帶入與查詢**：填寫新生資料成功或完成預約後，再次存取 `/booking/search`，系統應能從 `localStorage` 自動讀取並帶入最近使用者姓名，且自動重新跳轉完成查詢。
- `[ ]` **清除記憶與切換**：點擊「改用其他姓名查詢」按鈕，系統應清空 `localStorage` 記憶與姓名輸入框並返回空白查詢頁。
- `[ ]` **個資保護**：查詢結果只能顯示符合該學員身分的預約，不得查閱其他學員的隱私資料。
- `[ ]` **取消預約**：點擊「取消預約」能成功取消，且取消狀態同步回傳至資料庫。
- `[ ]` **逾期取消限制**：過期或已鎖定時段（開課前 7 天），「取消預約」按鈕必須不可用或予以限制。

---

## 6. 新生資料填寫流程

- `[ ]` **表單完整性**：進入 `/new-student`，確認「姓名」、「身分證字號」、「手機」、「生日」皆為必填項目，且含有個資同意告知條款。
- `[ ]` **個資勾選校驗**：不勾選「個資同意使用」直接送出，必須被擋下，並導向 `?error=consent` 顯示「請先勾選個資使用同意後再送出。」。
- `[ ]` **身分證字號與衍生末三碼**：填寫完整身分證字號送出，後端應自動擷取最後 3 碼作為學員的 `idNumberLast3` 並寫入資料庫。
- `[ ]` **Honeypot 防垃圾送出**：模擬機器人填寫表單中的 `website` 欄位（隱藏欄位），系統應默默將其重導向至 success 頁面，但不將垃圾資料寫入資料庫。
- `[ ]` **成功頁個資保護**：跳轉至 `/new-student/success` 後，畫面上絕不可顯露身分證字號、手機或生日等任何敏感個資。

---

## 7. 後台待確認新生審核

- `[ ]` **待確認新生名冊**：秘書處人員進入 `/admin/students?status=review`，能正確過濾出由自填入口送出（`source === "新生自填入口"` 且 `needsReview === true`）的待審核名單。
- `[ ]` **資料完整性核對**：點擊新生資料，確認身分證字號與自帶的末三碼皆正確無誤。
- `[ ]` **轉為正式學員**：確認資料完整後，可將其狀態由「待確認」改為「啟用」狀態。

---

## 8. 講師 / 助教授課入口

- `[ ]` **講師姓名登入**：進入 `/teaching/login`，輸入講師姓名，確認是否能順利登入。
- `[ ]` **共用授課通行碼**：
  - 如果環境變數中設定了 `TEACHING_ACCESS_CODE`，確認登入頁是否出現「授課通行碼」輸入框。
  - 通行碼輸入錯誤時，必須導回 `error=invalid`，無法進入授課中心。
  - 通行碼正確時，可順利進入並查看負責課程。
  - 所有連結切換（切換月份、切換 view 視角、點擊單堂）與 Server Action 重導向，通行碼 `code` 均能隨 URL query string 正確傳遞，不會產生點名或填寫日誌中途被踢出登入的狀況。

---

## 9. 點名與出席狀態

- `[ ]` **查看負責課堂**：講師只能查看自己負責的課堂日曆。嘗試存取非負責課堂（直接改網址 sessionId），系統應顯示「沒有授課權限」，不允許點名。
- `[ ]` **點名操作**：
  - 勾選出席、遲到、請假、缺席，狀態能即時更新並儲存。
  - 測試「一鍵全到」與「完成點名」，出席狀態同步更新至 Firestore。
- `[ ]` **課堂日誌與異常提報**：填寫「教學內容」、「異常狀況」並儲存，狀態能正確寫入。

---

## 10. 匯出與 Google Sheets 同步

- `[ ]` **資料導出**：進入 `/admin/exports`，測試導出學員名冊 CSV / Excel，確認格式正常且可順利下載。
- `[ ]` **Google Sheet 備份同步**：點擊「同步至 Google 試算表」，確認後台顯示成功。登入公司 Google 帳號檢查對應 Sheet ID（`1ZLGFID_8PDRfxgC8FSYoKGd3dM7BX8XBjhRnFoLPBis`）中的 12 張工作表是否皆已更新備份。

---

## 11. 手機版檢查

- `[ ]` **Responsive Layout**：
  - 學員前台 `/` 的三張分流卡片在手機（iOS/Android）上是否正常呈單欄直排，大按鈕是否容易點擊。
  - 新生自填 `/new-student` 的表單輸入框在手機上是否大小適中、不產生版面溢出。
  - 講師工作台 `/teaching` 及其點名頁面的日曆、點名控制鈕在手機上是否容易指尖操作。

---

## 12. 敏感資料與 GitHub 檢查

- `[ ]` **排除文件**：確認 `.env.local` 與 `firebase-admin-key.json` 不曾被 commit 至 Git 歷史紀錄中。
- `[ ]` **DEMO 資料去識別化**：確認 `data/booking-data.json` 中沒有包含真實學員的名字、真實手機、身分證字號與隱私預約個資。真實個資應僅儲存在雲端正式 Firestore。
